# Wayland Pointer Lock - 实施计划 (Phase 2)

## 方案对比

### 方案A: 修复API兼容性 (推荐) ⭐
**策略**: 降级gdk4-wayland到0.10.3匹配relm4
**优点**: 
- ✅ 版本对齐，API兼容
- ✅ relm4和gdk4-wayland使用相同的wayland协议版本
- ✅ wayland_crate feature正常工作
**实施**:
```toml
gdk4-wayland = { version = "0.10.3", features = ["wayland_crate"] }
```
**工作量**: 1小时

---

### 方案B: 使用Unsafe绕过 (临时) ⚠️
**策略**: 直接使用指针转换，跳过类型检查
**优点**: 快速实现
- ✅ 无需依赖版本对齐
**缺点**: 不安全
**实施**: 添加unsafe helper function
**工作量**: 30分钟

---

### 方案C: 运行时手动surface (中期) 🔧
**策略**: 在应用启动时从外部传入wl_surface
**优点**: 
- ✅ 完全绕过类型系统
- ✅ 程序架构清晰
- ✅ 用户可控
**缺点**: 需要修改example代码
**工作量**: 2-3小时

---

### 方案D: 条件编译 (长期) 📚
**策略**: 使用cfg属性分叉实现
**优点**: 
- ✅ 类型安全
- ✅ 编译时选择正确实现
- ✅ 代码整洁
**缺点**: 代码复杂度增加
**工作量**: 3-4小时

---

## 🎯 推荐方案

**优先级排序**：
1. 🥇 方案A (修复版本兼容) - **立即实施**
2. 🔧 方案B (运行时传递) - **备选方案**
3. ⚠️ 方案C (条件编译) - **长期方案**
